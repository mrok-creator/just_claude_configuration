# Customization

Everything project-specific is concentrated in two files inside the target
project: `.claude/cc-config.env` (stack behavior) and
`.claude/output-styles/comm.md` (language). Personal preferences go to
`.claude/settings.local.json`.

## cc-config.env reference

Auto-generated by `install.sh` from detection; edit freely. Hooks read it on
every invocation — changes apply immediately, no restart needed.

| Variable | Values | Effect |
|---|---|---|
| `CC_PKG_MANAGER` | `npm` \| `pnpm` \| `yarn` | The allowed package manager; the guard denies the other two |
| `CC_VALIDATE_MODE` | `nx` \| `command` \| `off` | Stop-hook validation: Nx touched-projects mode, a single command, or disabled |
| `CC_VALIDATE_CMD` | shell string | Used when mode=`command`, e.g. `npm run lint && npm run typecheck` |
| `CC_MIGRATION_GUARD` | `on` \| `off` | Blocks destructive migration/DB commands (typeorm revert, drop, truncate, unscoped delete) |
| `CC_MEMORY_NUDGE_REGEX` | regex or empty | Names in prompts matching this trigger a "query basic-memory first" nudge; empty disables |
| `CC_NAV_GUARD` | `on` \| `off` | Redirect symbol-shaped greps to Serena; turn off if Serena is not installed |
| `CC_NAV_GUARD_SUFFIXES` | `A\|B\|...` | Class-name suffixes treated as code symbols (defaults fit NestJS-style codebases) |
| `CC_NAV_GUARD_VERBS` | `a\|b\|...` | Leading verbs that mark camelCase method names as symbols |
| `CC_SOURCE_DIRS` | `dir1\|dir2` | Source-tree roots protected by read-guard/config-guard |
| `CC_PROTECTED_WRITE_DIRS` | `dir1/\|dir2/` | Dirs where bash file writes are hard-blocked (native tools only) |
| `CC_STATE_DIR` | path | Harness state home (default `.cc_settings`) |

## Language

Claude's conversation language is set in the PROJECT copy of the output style,
`.claude/output-styles/comm.md` (created by `/setup-project` — the project copy
exists precisely so plugin updates never reset your choice):

```
RESPONSE_LANGUAGE: English
```

Change the value to any language ("Ukrainian", "German", "Japanese", …) — the
rest of the style adapts automatically: prose in your language, code /
identifiers / commits / subagent prompts always in English. `/setup-project`
asks for this during installation; re-run it or edit the line directly to
change later. The change applies to new sessions.

## Plugin vs classic customization

Plugin installs: hooks/skills/agents live in the plugin cache — do NOT edit
them there (lost on update). Durable knobs are all project-side:
`cc-config.env`, `settings.json`, the output-style copy, `CLAUDE.md`. To change
the shipped behavior itself, fork the config repo and point your marketplace at
the fork. Classic installs vendor the files into `.claude/`, so editing them in
place is fine (the `editing-claude-config` skill guides safe edits).

## Personal settings (settings.local.json)

`settings.local.json` is gitignored and never shipped — model choice, personal
permission grants, context tuning. `install.sh` seeds it from
`settings.local.example.json`. The project-level `settings.json` deliberately
deny-lists Claude from editing `settings.local.json` — change it yourself.

Permission precedence reminder: **deny beats allow**, and rules merge across
`settings.json` (shared) and `settings.local.json` (personal).

## Disabling parts you don't want

| To disable | Do |
|---|---|
| Stop-hook validation | `CC_VALIDATE_MODE="off"` |
| Serena redirects | `CC_NAV_GUARD="off"` |
| Migration guard | `CC_MIGRATION_GUARD="off"` |
| Memory nudges | `CC_MEMORY_NUDGE_REGEX=""` |
| A whole hook | remove its entry from `.claude/settings.json` `hooks` section |
| /feature workflow | delete `.claude/commands/feature.md`, `.claude/workflow/`, and the role agents |
| An emergency bypass (one command) | prefix env: `READ_GUARD_OFF=1` / `NAV_GUARD_OFF=1` |

## Escalation / denial-cap tuning

Guards use a consecutive-denial cap (`.claude/hooks/lib/denial-cap.sh`):
identical repeated blocks escalate the message on the 2nd hit and, for
soft ("native-first") guards, auto-allow on the 3rd so the agent never
deadlocks. Tune with `DENIAL_CAP_N` and `DENIAL_CAP_WINDOW_SECONDS` env vars
in `cc-config.env` if needed.
